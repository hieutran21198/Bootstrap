# 0004. Adopt typed aggregate IDs with centralised UUIDv7 generation

- **Status**: Accepted
- **Date**: 2026-06-26
- **Deciders**: Minh Hieu Tran <hieu.tran21198@gmail.com>
- **Supersedes**: -
- **Superseded by**: -

## Context

[ADR-0003](0003-service-architecture.md) settled the broad service shape (DDD + CQRS + Hexagonal). One sub-decision inside that ADR — *how aggregates identify themselves and how those identifiers are produced* — is dense enough to deserve its own record. Three questions had to be answered together:

1. **Identifier typing.** Should `Organization.ID` be a raw `string`, a shared `ids.UUID`, or a per-aggregate typed alias (`type ID string`)? The portal service initially used raw `string`, and a near-miss during the staff/organization wiring (the owner-of-organization is a staff ID; the creator-of-staff is an organization ID — both `string` at the time) made the compile-time gap obvious.
2. **ID generation algorithm.** UUIDv4? KSUID? ULID? Snowflake? An auto-increment column? Each carries trade-offs around insert performance, sortability, format standardisation, and external dependencies.
3. **Where generation lives.** Each aggregate package could declare its own `NewID()` function, callers could inline `uuid.New().String()`, or one central helper could serve every aggregate. The first duplicates the algorithm; the second prevents the workspace from changing algorithms later; both leave the typed-vs-raw choice unenforced.

These choices interact: typed IDs are pointless if the generator returns raw `string`; centralising the generator only helps if the algorithm is fixed; and the algorithm choice constrains which database access patterns are sane (random UUIDv4 thrashes B-tree indexes, sortable formats don't).

The decision had to land before the portal service grew a second aggregate beyond `Organization` and `Staff`, because the cost of changing ID typing scales with the number of aggregates, repo conversions, and command handlers.

## Decision

We adopt **typed per-aggregate ID aliases, generated by a shared `idgen` package using UUIDv7**:

1. **Each aggregate owns a `type ID string` in its own `id.go`.** The alias is declared inside the aggregate's domain package (`organization.ID`, `staff.ID`, etc.). The package name qualifies the identifier; a method `String() string` implements `fmt.Stringer` for log/format use.

2. **Generation goes through [`bootstrap/packages/go/idgen`](../../packages/go/idgen).** The package exposes a generic helper `NewFor[T StringID]() T` parameterised by the typed alias: `idgen.NewFor[organization.ID]()` returns an `organization.ID`, not a raw `string`. A constraint interface `StringID = ~string` covers every per-aggregate alias. A companion `Validate[T StringID](id T) error` checks the well-formedness of an externally-supplied ID at system boundaries (HTTP handlers, queue consumers).

3. **The underlying format is UUIDv7** ([RFC 9562 §5.7](https://www.rfc-editor.org/rfc/rfc9562.html#section-5.7)) via [`github.com/google/uuid`](https://github.com/google/uuid) v1.6+ (`uuid.NewV7`). UUIDv7 embeds a millisecond Unix timestamp in the high bits, leaving the rest random — k-sortable, B-tree-friendly, RFC-standard.

4. **Cross-aggregate references stay as raw `string`** inside aggregate structs (`Staff.organizationID string`, `Organization.ownerStaffID string`). Typing them would require importing the foreign aggregate's package, which produces circular imports between sibling domain packages. The cost is mitigated by naming convention (`ownerStaffID`, never plain `id`) and an explicit cast at command-handler boundaries (`staff.ID(cmd.StaffID)`).

The rules above are the only places ID format is encoded in the workspace. A future ADR may change the algorithm; only `idgen` needs to change. A future ADR may relax (3) for a specific aggregate; the alias and the `idgen` API stay.

## Consequences

- **Positive**:
  - **Compile-time prevention of cross-aggregate ID swaps** for own primary keys. `func HireStaff(orgID organization.ID, ...)` cannot be called with a `staff.ID` value; the type system rejects it. The class of bug that motivated this ADR is closed at the call site, not at runtime.
  - **One place to change the algorithm.** If UUIDv7 stops fitting (e.g., a future requirement for shorter URL-safe IDs), only `packages/go/idgen` changes. No aggregate package depends on `github.com/google/uuid` directly.
  - **k-sortable primary keys.** UUIDv7's leading timestamp means sequential inserts hit contiguous B-tree pages — postgres index maintenance stays cheap. This is the single biggest reason to prefer UUIDv7 over UUIDv4 at scale.
  - **RFC-standard format.** UUIDv7 is parseable by any UUID library (`uuid.Parse` works without flags), formats as a normal hyphenated UUID, and is recognised by `psql`, GORM, OpenAPI tooling, and `pgcli`. No custom serialisation needed.
  - **Boundary validation is centralised.** `idgen.Validate(...)` lets HTTP handlers and queue consumers reject malformed IDs without each aggregate package writing its own parser; inside the domain, typed values are trusted because they have already passed the gate.
- **Negative**:
  - **Cross-aggregate references lose type safety.** `Staff.organizationID` is `string`, not `organization.ID`, so passing a misformed value compiles. The naming convention (`organizationID`) and the explicit cast at command-handler boundaries (`staff.ID(cmd.StaffID)`) are the only mitigations. This is the documented trade for not building a separate `ids` package.
  - **UUIDv7 carries a timestamp that is visible to anyone who parses the ID.** For internal aggregates this is fine; for IDs that escape to untrusted clients (e.g., a public API), the creation time of the resource is observable. Aggregates that need opaqueness (rare in this workspace) must override the default at design time.
  - **An extra package boundary.** Every aggregate import now pulls `idgen` for the generator. Acceptable cost — `idgen` is a 1-function stateless package with zero internal dependencies beyond `github.com/google/uuid`.
  - **The `uuid.Must(uuid.NewV7())` call panics on RNG failure.** OS-level random source failure is unrecoverable; we accept the panic over forcing every command handler to handle an `error` that essentially cannot occur. This is documented in `idgen`'s doc comment.
- **Neutral**:
  - **Builds on [ADR-0001](0001-single-responsibility-go-packages.md).** `idgen` is an SRP-stateless package, exactly fitting the stateless template — pure functions, no `Config`, no constructor.
  - **Builds on [ADR-0003](0003-service-architecture.md).** Typed IDs are the primary-key half of the aggregate root pattern documented there. This ADR is the deep dive that ADR-0003 defers to.
  - **Builds on [ADR-0002](0002-go-code-style.md) rule 5.** The `ID` type lives in `id.go` inside the consumer aggregate package — interfaces and types at the consumer, never in a separate `ids/` package.

## Alternatives considered

### Identifier typing

- **Raw `string` everywhere** (the status quo before this ADR). Rejected because cross-aggregate ID swaps compile cleanly. The portal had a real near-miss between `staff.ID` and `organization.ID` during owner-of-organization wiring; the compile-time guarantee from typed aliases eliminates that class of bug at zero runtime cost.
- **Shared `bootstrap/packages/go/ids` package** with `type OrgID string`, `type StaffID string`, ... centralised. Rejected because it breaks aggregate co-location — every domain package would import `ids/` for its own primary-key type, fragmenting the aggregate boundary. The aggregate is the unit of cohesion; its ID type belongs inside it.
- **Marker-interface based design** (`type ID interface{ aggregateID() }`). Rejected as over-engineering for a workspace that needs string IDs in a database. The marker interface adds zero practical guarantee that a typed alias doesn't already provide, costs an indirection at every call site, and complicates JSON / GORM mapping.

### ID generation algorithm

- **UUIDv4** (random 122 bits). Rejected because random insertion order causes B-tree page splits at scale — every insert touches a random leaf page, defeating sequential-write optimisations and inflating index size. UUIDv7 keeps the same uniqueness guarantees while making inserts append-mostly.
- **UUIDv1** (timestamp + MAC address). Rejected because (a) the MAC address leaks host identity, (b) the byte layout is little-endian timestamp segments interleaved with the clock sequence — *not* k-sortable as stored, requiring custom byte-reordering ("UUIDv1 with timestamp-first") to be index-friendly. UUIDv7 was specifically designed to fix this.
- **KSUID / ULID** (k-sortable, custom format). Rejected because both encode in a non-UUID format (27-char base62 for KSUID, 26-char base32 Crockford for ULID), so they don't fit a standard UUID column and don't render as a familiar `xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx` form. UUIDv7 delivers the same k-sortable property with the standard format that every tool already understands. Neither is in stdlib or in `google/uuid`; adding a dependency for non-RFC formats is a step back from the standardisation UUIDv7 provides.
- **Twitter Snowflake / instagram-style 64-bit IDs.** Rejected because they require coordinated worker IDs (machine number assignment), which is operational overhead with no payoff for a workspace where UUIDv7 already gives us k-sortable IDs without coordination.
- **MongoDB ObjectID** (12 bytes: timestamp + machine + counter). Rejected as Mongo-coupled — the format is recognised by Mongo tooling and rarely by anything else. We use postgres, not Mongo.
- **Auto-increment database column.** Rejected because (a) IDs cannot be generated pre-insert in command handlers without a round-trip to the database, breaking the UoW pattern where aggregates are constructed in memory with their final ID; (b) sequential integer IDs leak business cardinality (total customer count, user growth rate) when they escape to clients; (c) merging tenants or sharding becomes harder because IDs collide across shards. UUIDv7 keeps the timestamp ordering benefit without the leakage or coordination cost.

### Where the generator lives

- **Per-aggregate `NewID()` function** in every domain package. Rejected because (a) each aggregate would duplicate the `github.com/google/uuid` import and the algorithm choice; (b) changing the algorithm later requires editing every aggregate; (c) the duplicated code provides no additional safety over a single generic helper. The generic `idgen.NewFor[T]()` returns the typed alias and centralises the dependency in one package.
- **Inline `uuid.New().String()` at every call site.** Rejected because (a) it returns `string`, defeating the typed-ID design — every caller would need an explicit cast like `organization.ID(uuid.New().String())`, which is verbose noise that obscures intent; (b) callers can accidentally use `uuid.New()` (UUIDv4) instead of `uuid.NewV7()`, mixing algorithms across the codebase; (c) the workspace has no central place to swap the algorithm.
